# Internationalization (i18n) — design

Date: 2026-06-26
Issue: #9 (feat: Internationalization — French & English, with CI key-consistency check)

## Goal

Serve the amadis UI in French and English. Move every user-facing string out of
components into per-locale message catalogs. Resolve the active locale at runtime,
auto-detecting the visitor's browser language and letting them override it with a
toggle. Add a CI job that fails when the two catalogs drift apart.

## Decisions

- **Library**: `next-intl@^4.13.0`. Peer deps allow Next `^16.0.0` and React `^19.0.0`,
  so it is compatible with the project's Next 16.2.9 / React 19.2.7. It supports
  locale resolution without locale-in-URL routing, which is what we want.
- **Locales**: `en` and `fr`. The default and final fallback locale is `en`, because
  the existing UI copy is authored in English.
- **Locale resolution order** (highest priority first):
  1. `NEXT_LOCALE` cookie (set when the visitor uses the language toggle).
  2. `Accept-Language` request header, matched against `['fr', 'en']` (browser
     auto-detection).
  3. `en` fallback.
- **No URL routing**: locale never appears in the path. Pages keep their current routes.
- **CI parity check**: an in-order mirror check. Every non-reference locale must contain
  exactly the same keys as the reference locale (`en`), in the same order, nested
  structure included. Missing keys, extra keys, and order divergence each fail the build.

## Architecture

### Locale config — `i18n/config.ts`

Single source of truth shared by the app and the CI script:

```ts
export const locales = ['en', 'fr'] as const;
export const defaultLocale = 'en';
export type Locale = (typeof locales)[number];
```

### Request config — `i18n/request.ts`

next-intl's `getRequestConfig`. Resolves the locale using the order above, reading the
cookie and `Accept-Language` from `next/headers`, then loads
`messages/${locale}.json` and returns `{ locale, messages }`. A small header-parsing
helper picks the best match from `Accept-Language` against `locales`, defaulting to
`defaultLocale` when nothing matches.

### Plugin wiring — `next.config.ts`

Wrap the existing config export with `createNextIntlPlugin('./i18n/request.ts')`.

### Root layout — `app/layout.tsx`

Becomes `async`. Sets `<html lang={await getLocale()}>` and wraps `children` in
`<NextIntlClientProvider>` so client components can call `useTranslations`. The metadata
moves from the static `metadata` export to a `generateMetadata` function that reads
`getTranslations('home.metadata')`.

### Catalogs — `messages/en.json`, `messages/fr.json`

Nested JSON, one object per namespace. `en` is authored first and is the reference for
the parity check. `fr` mirrors it key-for-key. Namespaces: `common`, `nav`, `home`,
`auth`, `textes`, `dashboard`, `texteDetail`, `annotator`, `scanUploader`, `errors`.

### Reading strings in components

- Server components (`app/page.tsx`, textes list, dashboard, texte detail, `(app)/layout.tsx`):
  `const t = await getTranslations('namespace')`.
- Client components (`app-shell.tsx`, `theme-toggle.tsx`, `auth` pages, `annotator.tsx`,
  `scan-uploader.tsx`): `const t = useTranslations('namespace')`.
- Rich text (`annotator.folio.eyebrow`, `home.hero.lede`): `t.rich(...)` with a `<b>` tag
  rendered through the message, kept as one translatable unit instead of concatenated
  literals.
- Plurals (`textes.corpusCount`, `textes.list.annotationCount`, `dashboard.scholarCount`,
  `annotator.foot.codePoints`): ICU `{count, plural, ...}` messages.

### Server-action error strings — `errors` namespace

Server actions currently `throw new Error("literal")`. They will resolve the message at
the throw site with `await getTranslations('errors')`, since `getTranslations` works in
the server-action context and needs no extra plumbing. This keeps the change local to
each action. Coded errors translated at an error boundary are out of scope.

### Language toggle — `app/lang-toggle.tsx` + server action

A small client component placed beside `ThemeToggle` in `app-shell.tsx` and in the home
page top bar. It shows the inactive language and calls a server action that sets the
`NEXT_LOCALE` cookie and calls `revalidatePath('/')` so the new locale takes effect.
Theme stays in `localStorage` as it is purely client-side. Locale needs server-side
resolution, so it uses a cookie. The two switchers stay independent.

## CI parity check

### Script — `.github/scripts/check_translations.py`

Dependency-free Python 3 (standard library only), mirroring the MemorChess script of the
same name. Steps:

1. Discover locales by globbing `messages/*.json`. The reference locale is `en`
   (hardcoded constant in the script, matching `defaultLocale`).
2. Load each `messages/<locale>.json` (`json.load`, which preserves declaration order).
3. Flatten each catalog to an ordered list of dot-path keys (depth-first, preserving the
   JSON declaration order).
4. For each non-reference locale, compare its ordered key list to the reference (`en`):
   report keys present in the reference but missing in the locale, keys present in the
   locale but absent from the reference, and the first position where the order diverges.
5. Print a clear, grouped diff naming the offending locale and keys. Exit non-zero on any
   discrepancy, zero when every locale is an exact in-order mirror.

### npm script

`package.json`: `"i18n:check": "python3 .github/scripts/check_translations.py"` so it can
be run locally with the same command CI uses.

### CI job — `.github/workflows/ci.yml`

A new `i18n` job that runs in parallel with `lint`, `typecheck`, and `test`. It only needs
`actions/checkout` plus the runner's preinstalled Python 3 (no `npm install`, no
`prisma generate`, no `setup-python` since ubuntu-latest ships Python 3), so it is fast.
It runs `python3 .github/scripts/check_translations.py`. Add `i18n` to the aggregate
`check` gate's `needs` list so branch protection covers it.

## Tests (Vitest)

- `i18n/request.ts` resolution: cookie wins over header, `Accept-Language` is honored when
  no cookie is set, and `en` is the fallback when neither matches.
- The language-toggle server action sets the `NEXT_LOCALE` cookie.
- Updated component tests assert translated text renders through the provider for both
  locales where a test already exercises a string.

The Python parity script is verified out-of-band from Vitest (it is not Node). The script
carries a `--self-test` mode that builds matching and mismatching catalog fixtures in
memory and asserts the exit code and message for: a matching pair (pass), a missing key, an
extra key, and a reordered key (each fails). `python3 .github/scripts/check_translations.py
--self-test` runs it; no pytest or third-party dependency is added.

## Out of scope

Annotation offset logic stays language-agnostic. No locale-in-URL routing. No
runtime locale negotiation beyond cookie plus `Accept-Language`. No translation of stored
user data or the Old French specimen content.
